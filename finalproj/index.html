Abstract

In this project, we present a novel Bezier curve rasterizer designed for displaying TrueType files using a CPU-based text rendering system. Our rasterizer processes these curves to generate text in any font using TrueType files and the CPU's processing power. Additionally, the system incorporates a user-friendly graphical user interface that enables intuitive interactions, such as adjusting font size and font color. Unlike conventional font drivers, our renderer is capable of mixing conic and cubic Bezier curves in a single contour. Overall, this project demonstrates the potential of CPU-based rendering in digital typography by providing an efficient and accessible method for accurately rendering TrueType files.

Technical approach

In our font rendering project, we developed a custom solution for rendering text from TrueType files by interpreting fonts as glyphs and representing the required characters through them. TrueType is a widely-used font file format that includes glyph outlines defined by Bezier curves [4, 11]. Our approach focuses on rendering these curves by stitching them together and plotting the points.

Since we had implemented Bezier curve rendering in Project 2, we used Project 2 as the base of our project [3]. We decided to use FreeType, a library for font rendering, to work with TrueType files [10]. Since FreeType libraries are already provided by our CMake files, a simple include statement made them available.

Unfortunately, Project 2 could only render one Bezier curve. There was a need to create an alternate Renderer that would interpret the glyphs provided by our TrueType files, match them to the text to render, create the Bezier curves, and call the render method for each BezierCurve. This was accomplished by our Glyphs class which was set to be the Renderer used by the Viewer class in main.cpp.

Our Glyphs class is provided with the string of text to render and face of the FreeType font through the setText() and setFace() methods, and then calculates the curves by parsing the control points of each contour. These control points are available through FreeType’s outlines, and each point is simply classified as being “on” or “off” the curve [5]. Thus, we iterate through the control points, adding each one to the Bezier curve being currently processed. When we come across an “on” point, it is added as an endpoint to the current Bezier curve and the curve is added to an internal list, while a new Bezier curve is created. Within a single contour, successive Bezier curves are continuous, so this “on” point is also used as the beginning of the next Bezier curve. The last Bezier curve does not have an ending “on” point; instead, it uses the first point of the current contour as its endpoint. Given that we have parsed the control points into the Bezier curves correctly, we can trust the BezierCurve class implemented in Assignment 2 to render it properly.

Some of our features involved additional steps. In order to provide a range of font options, we included TrueType files we downloaded from Google Fonts and FontSpace [6, 12]. In order to provide color randomization features (such as setting the background to be a random color), we used C++’s <random> library [1, 8]. To change aspects of the appearance, such as the background color and the display text, we used OpenGL [2, 9]. We also activated multisampling in OpenGL to reduce aliasing [7].

Our approach differs from existing methods in the way we handle glyphs and the rendering process. Instead of relying on pre-rendered bitmap fonts, we opted for a more flexible and dynamic solution that allows for real-time rendering of text displayed based on TrueType files. This enables users to make adjustments on-the-fly, without the need for pre-rendered assets. Furthermore, unlike existing FreeType drivers, we can render both conic and cubic Bezier curves in a single contour, allowing greater flexibility and creativity in designing fonts for our renderer as opposed to conventional FreeType rendering.

Description of problem-solving

One of the main challenges we faced was adapting the existing Project 2 code to support multiple curves. To overcome this issue, we had to delve deep into the codebase and understand the underlying logic and structure. By doing so, we were able to discover the curves were plotted to the screen with the BezierCurve’s render() function, and create a Glyph class that would create individual BezierCurves and use each one’s render() function in turn to display each one to the screen.

Another challenge was integrating our rendering system with the graphical user interface (GUI). We had to understand how the existing code interacted with the GUI and determine the best approach to manipulate elements within the interface. The key discovery was that members of the Renderer class had a key_event() method that would respond in the case of an entered key, which we could modify in our Glyph class to respond to the user typing.

Lessons learned

Throughout this project, we learned several valuable lessons that can be applied to future endeavors. First, we gained a deeper understanding of the TrueType font format and the intricacies of glyph rendering. This knowledge will be invaluable for future projects in digital typography and text rendering.

We observed that the FreeType documentation is dense and hard to understand. This demonstrated to us that, even if software is widely used, it might not be easy to learn how to use it to perform basic tasks. This may be especially true for software like FreeType, which is designed to accommodate a wide variety of font formats and uses.

Additionally, we learned the importance of understanding a codebase before attempting modifications. By taking the time to comprehend the underlying logic and structure of the Assignment 2 code, we were able to make the necessary adjustments to support our font rendering system. Although we took steps to understand the codebase, there were some parts of it that were difficult to get a grasp on. We tried editing the CMakeLists to integrate NanoGUI, but got stuck. This made us realize that, even in a familiar codebase in a familiar language, dealing with dependencies and libraries can easily get complicated.

Finally, we learned that integrating a rendering system with a GUI requires careful consideration and planning. By understanding the interactions between the code and the interface, we were able to create a seamless user experience that allowed for real-time rendering of TrueType files. This experience has reinforced the importance of considering user interactions when developing software, ensuring that our solutions are both functional and user-friendly.

Results

We successfully accomplished our goals of implementing a CPU-based bezier curve rasterizer for fonts. We added many features to the GUI, including text translation, multiple font options, color changing, and timing features. By gaining a deeper understanding of the Assignment 2 code, we were successfully able to edit the GUI and generalize the single Bezier curve implementation in bezierCurve.cpp to the output of the FreeType library (glyphs).

First Iteration:

With added features:

Here, we showcase the text ‘graphics’ on the screen. This is done using the FreeType library’s functions to process TrueType files. With Project 2 code, we were able to process .dae and .bzc files, so we modified the handler to process TrueType files instead. Our program executable takes in a filepath of a font (we include Times New Roman, Open Sans, Roboto, Lato, Raleway, and Seaside as options). Then, a message is printed to the console asking the user for text to display. Once they enter text, the program processes it and displays it on the viewer. At the bottom left, you can see the amount of time taken to render the text, display the GUI, and total amount of time taken. Moreover, we have the time elapsed displayed on the bottom right of the GUI. The text size can be increased (+) and decreased (-); the text can be translated up (↑), down (↓), right (→), and left (←); and its color can be changed (‘R’, ‘G’, ‘B’). The color of the background can also be changed to a random color, as can the color of the outline of the text. Each curve can also be set to a random color simultaneously. We have all of these key commands available for display on the GUI by pressing ‘L’ to display a legend.
